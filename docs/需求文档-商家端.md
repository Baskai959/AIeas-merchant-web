# 商家/主播端 需求功能与开发任务文档（3 周 MVP）

> 端侧定位：PC 管理后台，面向**商家 / 主播 / 运营**角色，承担「商品上架 → 规则配置 → 开拍 → 控场 → 落锤 → 成交查看」的全流程操作。本端是「实时竞拍大师」业务闭环的发起方与控场方。
>
> 技术栈：React 18 + TypeScript + Zustand + Arco Design Pro + socket.io-client。

---

## 1. 角色与目标

| 维度 | 内容 |
| --- | --- |
| 主要使用人 | 商家 / 主播 / 运营 |
| 设备形态 | PC 浏览器（Chrome 优先） |
| 核心目标 | 在 3 周内交付一个**可上架、可控场、可落锤、可查单**的商家端 MVP |
| 与用户端的关系 | 商家端发布拍品并实时控场，用户端实时出价；通过 WebSocket 状态共享 |
| 与后端的关系 | REST 提交配置与命令；WebSocket 接收实时拍卖状态变化 |

## 5. 验收标准

| 类别 | 标准 |
| --- | --- |
| 功能完整性 | M-01~M-12 全部可用 |
| 实时性 | 控场面板价格/排名延迟 < 500ms |
| 一致性 | 落锤幂等，不出现重复成交 |
| 可用性 | 异常断线后自动重连并恢复控场状态 |
| 容错性 | 后端异常时给出明确提示，不出现白屏 |
| 加分（可选）| AI 文案 / 起拍价建议 / TTS 至少落地 1 项可演示 |

## 6. 与其他端的协作接口

### 6.1 调用后端 REST

| 接口 | 方法 | 说明 |
| --- | --- | --- |
| `/api/v1/auth/login` | POST | 登录 |
| `/api/v1/items` | POST/GET | 商品管理 |
| `/api/v1/auctions` | POST/GET/PATCH | 拍品管理 |
| `/api/v1/auctions/{id}/start` | POST | 开拍 |
| `/api/v1/auctions/{id}/hammer` | POST | 手工落锤（带 Idempotency-Key） |
| `/api/v1/auctions/{id}/cancel` | POST | 取消异常竞拍 |
| `/api/v1/orders` | GET | 成交订单 |
| `/api/v1/ai/start-price-suggest` | POST | 起拍价建议（加分项） |
| `/api/v1/ai/announcement` | POST | 文案生成（加分项） |

### 6.2 订阅 Socket.IO 事件

订阅房间：`auction:{auctionId}:control`

| 事件 | 用途 |
| --- | --- |
| `auction.started` | 拍卖开启 |
| `bid.accepted` | 出价被接受 |
| `ranking.updated` | 实时榜单 |
| `timer.tick` / `timer.extended` | 倒计时同步 / anti-sniping 触发 |
| `auction.closed` | 落锤完成 |
| `order.created` | 成交单生成 |

## 7. 风险与注意事项

1. **规则修改边界**：必须严格按状态拦截，禁止竞拍中修改规则；
2. **手工落锤幂等**：必须传 `Idempotency-Key`，防止重复点击造成重复成交单；
3. **倒计时漂移**：与服务端时间对齐，避免本地时钟误差导致显示偏差；
4. **断线恢复**：重连后必须重新拉取拍品当前状态，不依赖事件重放。
